# Taichi Long-Run Stability & Parity Recipe

---

## 1. Purpose

Document the workflow used to eliminate NaNs and achieve parity between the Taichi and NumPy backends during 16 k-step curvature-invariant runs. This serves as a playbook for future long-horizon experiments (topology sweeps, transport laws, etc.).

---

## 2. Symptoms & Detection

- NaNs appeared in `dissonance` after ~600–1,200 steps during long Taichi runs with default settings (`timestep=0.02`, `relaxation_rate=0.01`, `max_workers=9`).
- Telemetry (`runs/*/telemetry.csv`) showed rapid dissonance growth followed by `nan` values; plateau halting triggered early.
- NumPy backend exhibited the same behaviour under the high-energy initial conditions, confirming the issue was scenario-induced rather than Taichi-specific.

---

## 3. Stabilization Steps

1. **Lower timestep**: reduce `timestep` from `0.02` to `0.01` in long-run scenarios.
2. **Softer initial conditions**: scale random seed amplitudes (e.g., `psi_amplitude: 0.1`, `chi_amplitude: 0.02`) to avoid early blow-ups.
3. **Reduce workers**: `max_workers: 6` lowered peak dissonance while keeping throughput acceptable.
4. **Taichi backend tuning**:
   - `inner_iterations: 1` (limits per-launch amplification).
   - `relaxation_rate: 5e-4` (orders of magnitude lower than default 0.01).
   - `tile_shape: [8, 8]`, `use_vectorized: true` retained for performance.
5. **Halting criteria**: disable success-triggered stops for diagnostic runs; once stable, reinstate success checks but remove aggressive plateau halting (`dissonance_plateau_steps = 20000`, `dissonance_plateau_tolerance = 1e-5`).

The resulting configuration is stored in `scenarios/discovery_phase2/curvature_invariant_longrun.yaml`.

---

## 4. Verification Procedure

1. **Single-seed checks**
   - Run `Testing_scripts/discovery_phase2/run_curvature_invariant_ensemble.py` for 1 seed × 16 k steps using the long-run scenario to confirm clean telemetry and `discovery_verified = 1`.
2. **Parity**
   - Execute `Testing_scripts/run_taichi_parity_smoke.py` to ensure the stabilized Taichi configuration still matches NumPy outputs on shorter runs.
   - Optional: run a short NumPy job with matched timestep/amplitudes to verify parity without Taichi-specific adjustments.
3. **Ensemble sweeps**
   - Use `Testing_scripts/discovery_phase2/run_topology_probe_ensemble.py --scenario discovery_phase2/curvature_invariant_longrun.yaml` to generate 5×16 k runs per boundary.
   - Inspect `runs/discovery_phase2/curvature_invariant_topology_probe_16k.csv` for residual RMS, variance, and streak metrics.

---

## 5. Outcome

- Long-run torus/open/cylinder ensembles complete 16 k steps with residual RMS ≈ 1.4–1.9×10⁻³, invariant variance ≈ (1.8–3.4)×10⁻⁶, and verification streaks ≥ 178.
- Long-run evidence is archived in `runs/discovery_phase2/curvature_invariant_topology_probe_16k.csv`.
- The stabilized configuration serves as the template for future high-iteration Taichi jobs.

---

## 6. Future Recommendations

- For new scenarios, start from the long-run YAML and only increase relaxation or timestep after parity tests confirm stability.
- Extend this recipe to PR‑0 transport and other Phase II experiments before launching multi-topology sweeps.
- Consider automating stability checks via a CI job that runs a 2 k-step Taichi job with the long-run parameters and asserts `discovery_verified` remains 0 until expected thresholds.
