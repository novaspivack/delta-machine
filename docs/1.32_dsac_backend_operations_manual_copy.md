# DSAC Backend Operations Manual

**See also**: [dsac_users_guide.md](./dsac_users_guide.md) · [1.34_taichi_nan_parity_recipe.md](./1.34_taichi_nan_parity_recipe.md)

---

## 1. Purpose

Provide implementation guidance for engineers building or extending DSAC scenarios now that the Taichi backend is the production path. This manual captures backend selection rules, kernel configuration knobs, regression-offload policy, and parity/benchmark obligations.

---

## 2. Backend Selection & Entry Points

- **Core modules**  
  - Constraint backends: `src/delta_machine/backends/__init__.py`
  - Constraint DSL translator: `src/delta_machine/backends/dsl.py`
  - Scenario runner (regression offload, metrics): `src/delta_machine/scenarios/runner.py`
  - CLI (backend override): `src/delta_machine/cli.py`

- **CLI usage**  
  `python -m delta_machine.cli headless --scenario <yaml> --steps N --backend {auto|numpy|taichi}`  
  - `auto` (default) selects Taichi if `constraint_backend` metadata requests it; otherwise NumPy.  
  - Use `--backend numpy` for parity validation or debugging.  
  - Use `--backend taichi` to force Taichi even if the scenario metadata is misconfigured; the run will fail fast if fusion is unsupported.

- **Programmatic usage**  
  `DeltaOrchestrator` accepts `backend="numpy"` or `backend="taichi"`; passing `None` or `"auto"` defers to scenario metadata. Supply a `backend_config` dict when constructing the orchestrator to override scenario-level Taichi options (same keys described below).

## 3. Taichi Backend Configuration

Scenario YAMLs declare backend intent inside `constraint_backend`. Example (from `scenarios/discovery_phase2/curvature_invariant.yaml`):

```yaml
constraint_backend:
  type: taichi
  arch: cpu
  fusion_policy: auto
  inner_iterations: 2
  parallelize: auto
  tile_shape: [8, 8]
  block_dim: 64
  use_vectorized: true
```

Supported keys (see `TaichiConstraintBackend` in `backends/__init__.py`):

- `type` / `backend` / `name`: must be `taichi` to activate the backend.
- `arch`: `cpu` is production. `gpu` works only with `dtype: float32` and remains slower; treat as experimental.
- `dtype` / `precision` / `fp_precision`: `float64` default, `float32` (or `float16`) optional when memory/throughput trade-offs dominate. The Taichi field dtype controls state buffers and relaxation rate scaling.
- `cpu_max_num_threads`: integer or `auto`; maps to Taichi runtime thread pool.
- `inner_iterations`: number of kernel inner loops executed per launch (amortises Python↔Taichi overhead). Keep ≥1; 2–4 is the sweet spot for discovery workloads.
- `parallelize`: integer or `auto`; sets Taichi `ti.parallelize` hint for outer loops. When omitted, Taichi auto-tunes.
- `block_dim`: power-of-two block size for tiled kernels. `auto` picks 64; backend rounds to nearest power of two.
- `tile_shape`: `[tile_y, tile_x]`, integer, list, or `auto`. Enables block-local caching plus vector lanes. The backend auto-derives a block_dim if absent.
- `use_vectorized`: `true` to stage derived quantities with `ti.Vector` lanes. Requires `tile_shape` with `tile_x ≥ vector_width`.
- `vector_width`: lane width for vectorised math (default 4). Backend falls back to scalar if tiling or width constraints are unmet.
- `fusion_policy`: `auto` (single fused kernel), `per-node` (one kernel per constraint), or omit to supply an explicit `fusion` list. Large fusions give best throughput once vectorized; use `per-node` during debugging.
- `fusion`: optional list of dictionaries describing custom fusion groups.
- `relaxation_rate`: overrides the global relaxation scalar mirrored in NumPy backend (default 0.01 for CPU float64, lower for float32/GPU).

**Guidelines**
- Keep `dtype: float64` + `arch: cpu` for production curvature/transport workloads (numerical stability, fastest path).  
- Set `inner_iterations: 2` to balance stability and launch cost; higher values may overshoot with aggressive relaxation.  
- Leave `parallelize: auto` unless profiling shows under-utilisation; manual thread counts should not exceed physical cores.  
- Use `tile_shape` + `use_vectorized: true` for metric-closure or discovery residual families; stick with scalar tiles when introducing new kernels to simplify debugging.  
- Prefer `fusion_policy: auto` for mature kernels; fall back to `per-node` if a new constraint misbehaves or parity fails.

## 4. Discovery Regression Offload Policy

Implemented in `src/delta_machine/scenarios/runner.py::_compute_discovery_metrics`.

- Accumulates `XᵀX`, `Xᵀy`, and `yᵀy` on the Python side; solves intermittently via `np.linalg.solve` (fallback `np.linalg.lstsq`).
- Metadata knobs:
  - `discovery_fit_interval` — minimum step spacing between solves (defaults to `max(sample_interval, 1)`).
  - `discovery_fit_min_samples` — minimum accumulated samples before first solve (default `feature_dim * 4`).
  - `discovery_ridge_lambda` — optional ridge penalty injected on the diagonal.
  - `discovery_accumulator_decay` — exponential decay (0–1) applied after each solve to forget stale data.
- Always request `discovery_sample_interval` and `discovery_burn_in_steps` to avoid fitting noisy warm-up windows.
- Leave regression inside Taichi only when a constraint requires fully fused Jacobian solves; otherwise prefer this offload path to keep kernels simpler.
- Transport runs now record additional diagnostics (`transport_mape_trimmed`, `transport_flux_divergence_*`, `transport_predicted_rms`) in scenario metrics and CSV outputs; include these in lab notes when reporting ensembles.

## 5. Scenario Authoring Conventions

1. **Metadata structure**  
   - Set `scenario_type: discovery` for law-finding runs so `_compute_discovery_metrics` is activated.  
   - Provide `constraint_backend` block per Section 3.  
   - Record analytic truth in `discovery_truth` (coefficient map) and `discovery_truth_expression` when available.

2. **Field coverage**  
   - Taichi backend currently recognises fields in `FIELD_VAR_MAP` inside `backends/__init__.py` (`psi_real`, `psi_imag`, `chi`, `chi_dot`, `dissonance`).  
   - Adding new fields requires updating `FIELD_VAR_MAP`, `FIELD_SNAPSHOT_MAP`, and ensuring kernels declare/assign those fields via the DSL translator.

3. **Constraint DSL**  
   - Symbolic expressions originate from `ConstraintGraph` nodes; the translator (`backends/dsl.py`) maps SymPy objects to Taichi code.  
   - Extend `KNOWN_SYMBOLS` (in `dsl.py`) for reusable derived terms before hand-writing kernel fragments.  
   - For complex non-polynomial behaviour, introduce a new symbolic helper instead of embedding long Python expressions directly.

4. **When to embed new Taichi kernels**  
   - Fuse via DSL whenever the constraint is expressible in SymPy/NumPy vocabulary (polynomials, logs, simple ratios).  
   - Embed bespoke Taichi code only if: (a) constraint relies on custom control flow, or (b) the DSL translator cannot represent it without massive expansion.  
   - Document any hand-crafted kernels in the `DSAC_tools/` directory or a new doc under `docs/` once added.

5. **Path conventions**  
   - Reference scripts and generators by their repo-relative path.  
   - Example: `Testing_scripts/discovery_phase2/run_curvature_invariant_ensemble.py`.  
6. **Further reading**  
   - See `docs/dsac_users_guide.md` for programmer workflows and `docs/dsac_discovery_recipe.md` for the discovery-to-proof checklist.

## 6. Parity, Testing, and Benchmark Workflow

- **Smoke parity**: `Testing_scripts/run_taichi_parity_smoke.py`  
  - Extend `DEFAULT_CASES` whenever a new scenario is Taichi-enabled.  
  - Ensure tolerances reflect the configured dtype (`_auto_tolerance` already adapts).

- **Benchmark loop**: `Testing_scripts/run_discovery_benchmark_suite.py`  
  - Use the 5×8 k suite for quick iteration; re-run the 20×16 k auto/per-node cases before promoting a kernel change to production.  
  - Archive CSV outputs under `runs/` with descriptive filenames.

- **Regression checks**: Every structural change to kernels or scenario metadata must demonstrate:  
  1. Parity suite passes (`--backend numpy` vs Taichi).  
  2. Benchmarks show non-regressive runtime (±5%).  
  3. Discovery verification streaks remain stable on canonical scenarios (polynomial, TE₁, PR-0, curvature).                                                                                              
- **Flux diagnostics**: Use `DSAC_tools/analyze_pr0_transport_flux.py` (plus `src/delta_machine/analysis/transport_flux.py`) to verify Laplacian/source decomposition when preparing proof packages.

## 7. Known Limitations & Warnings

- GPU Taichi (`arch: gpu`) remains slower (~4.6 ms/step vs 1.7 ms/step CPU) and requires `dtype: float32` plus lower `relaxation_rate` Document any GPU experiments explicitly.  
- Taichi runtime initialisation is global; differing `init_kwargs` across runs will raise an error. Use consistent configs within a process.  
- Vectorised kernels demand non-trivial parity updates; disable `use_vectorized` when bootstrapping new constraint families.  
- Regression offload currently assumes dense features; sparse representations need bespoke handling before enabling.

---

## 8. Quick Checklist for New Scenarios

1. Duplicate an existing Taichi-enabled discovery YAML as a template.  
2. Populate `discovery_features`, `discovery_truth`, tolerances, and backend metadata.  
3. Run NumPy baseline (`--backend numpy`) to confirm correctness.  
4. Switch to Taichi, collect parity traces, and benchmark.  
5. Record results in `runs/` with descriptive filenames.  
6. Add the scenario to parity + benchmark scripts.  
7. If kernels require new symbolic helpers, extend `backends/dsl.py`.

---

With these practices codified, contributors can extend DSAC without re-discovering Taichi integration pitfalls or regressing the validated discovery pipeline.
